# DevOps Automation Scripts

Этот документ описывает проект `deb_scripts` — набор скриптов автоматизации DevOps-задач в Linux-окружении.

## Обзор

`deb_scripts` — это набор Python-скриптов, предназначенных для автоматизации распространенных DevOps-задач в Linux-системах, основанных на apt (Debian, Ubuntu и их производные). Проект предоставляет удобное интерактивное меню для выполнения различных операций настройки сервера.

## Инструкции для AI

1. **Безопасность**: все изменения должны быть идемпотентными. Перед изменением системных файлов проверяй, что скрипт запущен с root-правами.
2. **Стиль кода**:
   - Типизация обязательна.
   - Следуй PEP 8 (макс. длина строки 99).
   - Имена: `snake_case` для функций и переменных, `PascalCase` для классов.
3. **Логирование**: используй `utils.logger` с эмодзи. Уровни: INFO для основных этапов, DEBUG для деталей, ERROR для проблем.
4. **Обработка ошибок**: перехватывай конкретные исключения, логируй и возвращай `False` при неудаче.
5. **Формат ответа**: предлагай изменения в виде диффов или полных файлов. Для новых функций сначала опиши архитектуру.
- **Тестирование**:
   - Все тесты должны запускаться в **DevContainer**, а не на локальной машине.
   - Используй **pytest** для написания и выполнения тестов.
   - Для запуска тестов используй команду: `pytest` внутри DevContainer
   - Подробнее см. раздел «Тестирование в DevContainer».
- **Интерактивный режим**:
   - Все интерактивные меню должны использовать эмодзи для улучшения UX.
   - Добавляй информационные панели с описанием инструментов и ссылками на официальную документацию.
   - Следуй единообразному формату: "номер - эмодзи - название действия".
   - Добавь команду 00 для отображения информации о каждом сервисе.

## Соглашения по коду

- **Сервисы**: класс с методами `install()`, `uninstall()`, `is_installed()`. Используют `subprocess_utils` для выполнения команд.
- **Интерактивные меню**: функция `show_<service>_menu()` вызывает соответствующие методы сервиса на основе выбора пользователя.
- **Утилиты**: общие функции вынесены в `utils`, дублирования кода быть не должно.
- **Режим разработки**: при `--mode dev` включается DEBUG-логирование, и скрипт может пропускать некоторые проверки (например, подтверждения).

## Тестирование в DevContainer

### Запуск тестов

```bash
# Запустить все тесты (внутри DevContainer)
pytest

# Для получения отчета о покрытии
pytest --cov=app --cov-report=term-missing

# Для запуска только юнит-тестов
pytest -m unit

# Для запуска только интеграционных тестов
pytest -m integration

# Для запуска только системных тестов
pytest -m system
```

### Структура тестов

Тесты размещаются в подкаталогах `tests/unit/`, `tests/integration/` и `tests/system/` и именуются `test_*.py`. 
- **Юнит-тесты** (`tests/unit/`): проверяют отдельные функции и классы изолированно, используют моки для зависимостей
- **Интеграционные тесты** (`tests/integration/`): проверяют взаимодействие между различными компонентами, используют метки `@pytest.mark.integration`
- **Системные тесты** (`tests/system/`): проверяют работу команд и функций в реальной системной среде, но изолированной через DevContainer
Используй pytest-фикстуры для настройки окружения. Поскольку скрипты изменяют систему, тесты должны быть идемпотентными и, по возможности, использовать моки для внешних вызовов. Однако для критически важных операций (установка Docker, включение BBR) можно выполнять реальные действия внутри контейнера — контейнер всё равно будет уничтожен после тестов.

### Примечания по тестированию

- DevContainer запускается с необходимыми правами, так как некоторые скрипты (BBR, Fail2Ban) требуют изменения системных параметров ядра или работы с systemd. Это безопасно только в изолированной среде.
- Все изменения, сделанные внутри DevContainer, не влияют на хост.
- Покрытие кода (coverage) обязательно для новых сервисов. После запуска pytest с флагом `--cov` отчёт выводится в терминал.
- Для тестирования интерактивного меню используй pytest с мок-вводом (например, monkeypatch) или expect.
- Покрытие кода должно быть не ниже 70% для новых сервисов.

## Важные конфигурационные файлы

- `pyproject.toml` — зависимости и настройки.
- `run.sh` — обёртка для запуска с проверкой Python.
- `app.log` — лог-файл (игнорируется git).
- `Dockerfile` — образ для DevContainer (основная среда разработки).
- `compose.yml` — конфигурация для запуска тестов в изолированной среде.
- `tests/conftest.py` — общие фикстуры и настройки pytest.
- `tests/README.md` — документация по структуре тестов.
- `tests/system/` — системные тесты для проверки работы в реальной среде.
- `.devcontainer/devcontainer.json` — конфигурация DevContainer для удобной разработки в изолированной среде.
- `interfaces/cli/non_interactive.py` — модуль для неинтерактивного выполнения команд с аргументами командной строки (например, `--install 1 2 3`).
- `Dockerfile` — использует `debian:bookworm-slim` в качестве базового образа.
- `.devcontainer/` — конфигурация для разработки в изолированной среде с предустановленными инструментами.

## Использование (для справки)

```bash
git clone https://github.com/thegrayfoxxx/deb_scripts.git
cd deb_scripts
# Для пользователей:
sudo ./run.sh          # или sudo python3 main.py
# режим разработки: sudo python3 main.py --mode dev

# Для разработчиков:
# Откройте проект в DevContainer (предпочтительно) или запустите в обычном окружении с установленными зависимостями
```
