# 📚 Документация deb_scripts

`deb_scripts` — набор Python-скриптов для автоматизации типовых DevOps-задач в Linux-окружении на базе `apt`.

## ✨ Что делает проект

Проект предоставляет единый интерфейс для:

- настройки `UFW`
- установки, включения и отключения `BBR`
- установки и удаления `Docker`
- установки, активации и удаления `Fail2Ban`
- установки и удаления `TrafficGuard`
- установки и удаления `uv`

Поддерживаются:

- интерактивный режим с меню
- неинтерактивный CLI
- операторский вывод в консоль
- логирование в `app.log`

Текущая версия проекта: `2.1.3`.

## 🧰 Системные требования

- `Python 3.12+`
- предустановленный `git`
- `Debian 12+` или `Ubuntu 22.04+`
- доступ к `apt`
- запуск от `root` для операций, меняющих системное состояние

Минимально поддерживаемые дистрибутивы:

- `Debian 12+`
- `Ubuntu 22.04+`

Проект рассчитан на запуск именно от `root`. Для обычного пользователя штатный способ запуска это `sudo ./run.sh`.

## 🚀 Установка и первый запуск

### Рекомендуемый способ

```bash
git clone https://github.com/thegrayfoxxx/deb_scripts.git
cd deb_scripts
sudo ./run.sh
```

Перед этим в системе уже должен быть установлен `git`, так как штатный способ установки начинается с `git clone`.

Если текущий пользователь уже `root`, запускай без `sudo`:

```bash
./run.sh
```

### Что делает `run.sh`

`run.sh` — основной launcher проекта. Именно через него рекомендуется запускать приложение.

На старте скрипт:

1. определяет корректный `HOME` эффективного пользователя
2. пытается подгрузить `"$HOME/.local/bin/env"`
3. проверяет наличие `uv`
4. если `uv` отсутствует, при необходимости устанавливает `curl`
5. устанавливает `uv` официальной командой `curl -LsSf https://astral.sh/uv/install.sh | sh`
6. повторно подгружает `"$HOME/.local/bin/env"`
7. создает собственные симлинки в `/usr/local/bin` для глобального вызова `uv`, `uvx` и `uvw`
8. выполняет `uv sync`
9. запускает приложение через `./.venv/bin/python3 ./main.py`

Такой путь нужен, чтобы:

- использовать стандартную установку `uv` в `~/.local/bin`
- одновременно обеспечить глобальный вызов `uv` при запуске через `root` или `sudo`
- не требовать ручной подготовки `.venv`

### Нужна ли ручная установка зависимостей

Нет, в обычном сценарии ручная подготовка окружения не нужна. `run.sh` сам выполняет bootstrap:

- устанавливает `uv`, если его нет
- синхронизирует проектное окружение через `uv sync`
- запускает `main.py` из локального `.venv`

Ручная работа с `uv` нужна только для разработки или отладки.

## 🔀 Режимы работы

Проект поддерживает два режима:

- интерактивное меню
- неинтерактивный CLI

Если `run.sh` вызывается без CLI-аргументов, запускается интерактивный режим.

Если переданы аргументы вроде `--install`, `--status` или `--info`, приложение работает в неинтерактивном режиме.

## 🖥️ Интерактивный режим

Штатный запуск:

```bash
sudo ./run.sh
```

Если пользователь уже `root`:

```bash
./run.sh
```

После запуска открывается главное меню со списком сервисов и их текущим статусом.

Главное меню использует такие коды:

- `1=UFW`
- `2=BBR`
- `3=Docker`
- `4=Fail2Ban`
- `5=TrafficGuard`
- `6=UV`

Для каждого сервиса доступно отдельное подменю с операциями установки, удаления, статуса, а для activatable-сервисов еще и активации/деактивации.

## 💻 Неинтерактивный CLI

### Поддерживаемые аргументы

- `--log-level {debug,info,warning,error}`
- `--install [codes...]`
- `--uninstall [codes...]`
- `--activate [codes...]`
- `--deactivate [codes...]`
- `--status [codes...]`
- `--info [codes...]`
- `--lang {ru,en}`
- `--all`

### Правила использования

- `--all` применяется только к одной явно выбранной операции, например `--status --all`
- нельзя одновременно передавать коды сервисов и `--all` для одной и той же операции
- если операция указана без кодов, нужно либо передать коды, либо использовать `--all`
- язык по умолчанию `ru`, для английского интерфейса используй `--lang en`

### Примеры

Для обычного пользователя запускай примеры через `sudo`. Если текущий пользователь уже `root`, убери `sudo`.

```bash
sudo ./run.sh --status --all
sudo ./run.sh --install 1 2 3
sudo ./run.sh --activate 1 2 4
sudo ./run.sh --info 1 4 6
sudo ./run.sh --lang en --status 1 4
sudo ./run.sh --log-level debug --uninstall 3
```

## 🧩 Поведение сервисов

### UFW

- установка и настройка `ufw`
- безопасный baseline: `deny incoming`, `allow outgoing`
- сохранение доступа по SSH
- отдельные операции активации и деактивации

### BBR

- настройка параметров ядра для `TCP BBR`
- проверка установленного состояния
- отдельные операции активации и деактивации

### Docker

- установка `Docker Engine`
- полное удаление компонентов Docker
- статус и справочная информация

### Fail2Ban

- установка `fail2ban`
- настройка и активация защиты SSH
- отдельные операции активации и деактивации

### TrafficGuard

- установка и удаление комплексной сетевой защиты
- статус и справочная информация

### UV

- установка современного Python package manager `uv`
- удаление `uv` и связанных данных
- статус, версия и проверка доступности из `PATH`

Для `uv` используется следующая схема:

- стандартная установка остается в `~/.local/bin`
- при запуске от `root` или через `sudo` проект дополнительно создает собственные симлинки в `/usr/local/bin`
- это позволяет сохранить дефолтный install flow `uv` и при этом дать глобальный вызов `uv`, `uvx`, `uvw`

## 📝 Логирование

Проект пишет операторские сообщения в консоль и, при возможности, в `app.log`.

Уровни:

- `debug` — подробная диагностика и внутренняя телеметрия
- `info` — нормальный операторский вывод
- `warning` — деградированные, но не фатальные ситуации
- `error` — ошибки операции

Пример:

```bash
sudo ./run.sh --log-level debug --status 6
```

## 🔄 Поведение при обновлении ОС

При запуске приложение не делает безусловный `apt update` на каждом старте.

Текущая логика:

- `apt update` пропускается, если последний успешный update был меньше `6` часов назад
- `apt upgrade` выполняется только если `apt list --upgradable` показывает реальные пакеты для обновления

Это поведение централизовано в точке входа приложения, а не размазано по отдельным сервисам.

## 🛠️ Разработка

### Локальная работа с `uv`

Если ты разрабатываешь проект локально и хочешь подготовить окружение вручную:

```bash

```

После этого можно запускать тесты напрямую:

```bash

```

- `run.sh` — основной launcher и bootstrap окружения

## 🏷️ Релизы

Версия проекта хранится в `pyproject.toml`.

Если изменение тянет на релиз:

1. обнови версию через `uv version --bump patch|minor|major`
2. создай git tag с той же версией в формате `vX.Y.Z`
3. запушь ветку
4. запушь тег

Пример:

```bash
uv version --bump patch
git tag v2.1.4
git push origin master
git push origin v2.1.4
```

Пуш тега, совпадающего с шаблоном `v*`, запускает GitHub Actions workflow, который создает GitHub Release с автоматически сгенерированными notes.

## 🆘 Типовые проблемы

### `uv` установлен, но не вызывается напрямую через `sudo uv`

Проверь, что launcher уже выполнялся и создал симлинки в `/usr/local/bin`. Штатный путь bootstrap это запуск через `run.sh`.

### Приложение не запускается без `sudo`

Это ожидаемо. Проект рассчитан на запуск от `root`, потому что управляет системными пакетами, firewall, sysctl и сервисами.

### Нужен ли прямой запуск `python3 main.py`

Для обычной эксплуатации нет. Рекомендуемая точка входа — `run.sh`, потому что он подготавливает `uv`, `.venv` и окружение запуска.
